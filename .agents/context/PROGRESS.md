- 2026-04-27 · _(this PR)_ · refactor(storage): dual storage backend (S3-compatible + Postgres blob) selectable via `STORAGE_BACKEND=s3|postgres` (default `s3`). `StorageService` is now a thin facade over a `StorageBackend` interface with two impls in `apps/api/src/storage/backends/`: `S3StorageBackend` keeps the existing ADR-0025 behaviour (Backblaze B2 / R2 / MinIO / AWS S3 via `@aws-sdk/client-s3`), and the new `PostgresStorageBackend` writes blobs to a fresh `StorageObject` Prisma model (`key`, `folder`, `filename`, `mimeType`, `size`, `bytes`, `createdAt`) as `bytea`. New `StorageController` serves them via `GET /api/storage/<folder>/<filename>` with full HTTP **Range** support — uses `SELECT substring(bytes from $start+1 for $length)` so a seek into a 5 MB MP3 only fetches the chunk it needs (TOAST out-of-line storage means range reads don't pay for the rest of the blob). Sibling admin endpoint `GET /api/admin/storage/info` reports `{ backend, objectCount, totalBytes }` and feeds the admin dashboard "Storage backend" stat card (now shows live disk usage + object count for the postgres backend). `docker-compose.yml` flipped to provision local Postgres + Redis services by default (was commented out, with cloud DBs assumed); operators on Neon/Supabase/Upstash drop the services and override the URLs. `.env.production.example` + `apps/api/.env.example` document both backends and add `API_PUBLIC_URL` (used by the postgres backend so upload URLs become `${API_PUBLIC_URL}/api/storage/<key>`). 109 unit tests pass; full typecheck/lint/build green. STACK / ARCHITECTURE / FEATURES / DECISIONS updated; ADR-0026 added (extends ADR-0025; both backends co-exist).

- 2026-04-27 · #27 · feat(admin): bulk operations on `/tracks`. Two new admin-guarded endpoints — `POST /api/admin/tracks/bulk-delete` and `PATCH /api/admin/tracks/bulk-genre` — both capped at 200 ids/request via `class-validator` (`ArrayMaxSize`). Both return a `{ deleted/updated, notFound }` partition: non-existent ids and non-upload tracks land in `notFound` instead of throwing, so a single bad row never blocks the batch. Bulk-delete is best-effort against R2 (errors swallowed, matching `remove()`); the DB `deleteMany` runs after the parallel `storage.delete` fan-out so the row vanishes even if the object lingers. Bulk routes are declared **before** the `:id` parameterized routes in the controller — NestJS matches in declaration order, so `@Patch(':id')` would otherwise swallow `/bulk-genre`. Admin `/tracks` page gains per-row checkboxes, a master "select all on this page" checkbox with `indeterminate` state via a ref callback, and a sticky floating action bar with `Set genre…` (dropdown of `GENRES` + "Clear genre") and `Delete` (with confirm dialog). Selection persists across pagination but is cleared on search-debounce so a filtered list can't accidentally bulk-edit hidden results. New `apps/api/src/admin-tracks/admin-tracks.service.spec.ts` covers the partition logic, R2-failure tolerance, dedup, the empty-input fast path, and the genre-clear case (7 tests). FEATURES updated; no ADR (extension of an existing admin surface, not a new architectural decision).
